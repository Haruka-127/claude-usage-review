# Claude Usage Review レポート形式

このドキュメントは、`claude-usage-review` Skillが生成するMarkdownレポートの形式を定義する。

レポートは、責任追及や点数付けではなく、次回から実行できる改善に焦点を当てる。

## 基本方針

- ログ上の根拠に基づいて書く。
- 短い引用や観測事実を含める。
- 秘密情報、長いコード断片、巨大なtool outputは引用しない。
- 過去ログ由来の分析と、現在のプロジェクト資産およびグローバル資産の分析を明確に分ける。
- プロジェクトまたはグローバルの`CLAUDE.md`やSkillが過去セッションに影響したとは断定しない。
- 各指摘は、次回から試せる具体的な行動に変換する。

## 保存先

レポートは、Skillを実行したカレントディレクトリの`reports/`配下に保存する。

収集スクリプトが生成したJSONも、同じ`reports/`配下に保存する。

推奨ファイル名:

- 収集JSON: `reports/claude-usage-review-input-YYYY-MM-DD.json`
- Markdownレポート: `reports/claude-usage-review-YYYY-MM-DD.md`

`YYYY-MM-DD`は実行日の日付に置き換える。

生成されるJSONとレポートには、redact済みであっても個人のClaude Code利用状況が含まれるため、不要にコミットしない。

## 必須セクション

レポートには以下のセクションをこの順序で含める。

1. 対象期間と分析対象
2. 全体サマリー
3. ログから見えたClaude Codeへの依頼の出し方
4. 効率よく進んだPrompt Episodeの特徴
5. 非効率になったPrompt Episodeの特徴
6. デバッグ依頼の改善点
7. 実装・リファクタ依頼の改善点
8. 現在のCLAUDE.md整備状況
9. 現在のSKILLS整備状況
10. 次回から使うプロンプトテンプレート
11. 次の1週間で試す運用改善

## 各セクションの内容

### 1. 対象期間と分析対象

以下を簡潔に記載する。

- 収集対象期間
- スキャンしたセッション数
- 抽出したPrompt Episode数
- Project Assetsとして分析したプロジェクト数
- parse errorやunknown eventなど、分析の信頼性に関わるメタ情報

### 2. 全体サマリー

ユーザーのClaude Code利用について、重要な傾向を3から5点で要約する。

良い点と改善余地の両方を含める。

### 3. ログから見えたClaude Codeへの依頼の出し方

Prompt Episodeから、依頼文の特徴を分析する。

見る観点:

- 初回依頼の具体性
- ファイル名や対象範囲の明示
- 達成したい作業内容の有無
- 期待結果、完了条件、受け入れ条件の明示度
- 検証コマンドの明示
- 出力形式、スコープ制約、再現手順、直近変更文脈の有無
- 短文承認の使い方

`has_task_goal`と`expected_result_strength`を分けて読むこと。依頼の目的が読み取れる場合でも、期待結果や完了条件が明示されているとは限らない。

`expected_result_evidence`には、期待結果を明示的と判断した根拠が入る。根拠が空で`expected_result_strength`が`implicit`の場合は、「作業目的はあるが完了条件は明示されていない」と解釈する。

`prompt_intent`は依頼の大まかな種類を示す補助情報であり、断定的な分類として扱わない。

`deliverable_type`は、依頼が求めている成果物の種類を示す補助情報である。`commit_message`、`report`、`summary`、`plan`、`review`、`template`、`code`、`unknown`のいずれかになる。`prompt_intent`と組み合わせて、「何をしてほしいか」と「何を成果物としてほしいか」を分けて読む。

### 4. 効率よく進んだPrompt Episodeの特徴

tool call数、toolカテゴリ、失敗の少なさ、検証状態、次のユーザー反応などから、効率よく進んだ例を挙げる。

短い引用と観測事実をセットで示す。

### 5. 非効率になったPrompt Episodeの特徴

探索過多、Bash失敗、同一コマンドの繰り返し、検証失敗または未検証、次ターンでの修正要求などを根拠に分析する。

ただし、ユーザーのプロンプトだけを原因として断定しない。

### 6. デバッグ依頼の改善点

デバッグ依頼で不足しがちな情報を整理する。

見る観点:

- エラーログ
- 再現手順
- 期待結果
- 直近で変更した内容
- 実行済みの検証コマンド

`has_reproduction_steps`、`has_recent_change_context`、`has_error_excerpt`、`verification_status`を参考にする。

### 7. 実装・リファクタ依頼の改善点

実装やリファクタ依頼で、スコープや完了条件が明確だったかを分析する。

見る観点:

- 対象ファイルや対象機能
- やること、やらないこと
- 既存仕様の扱い
- テストやlintの期待
- セッション分割が必要そうな大きな依頼

`has_scope_limit`、`has_acceptance_criteria`、`has_output_format`、`deliverable_type`、`files_read`、`files_edited`、`tool_categories`を参考にする。

### 8. 現在のCLAUDE.md整備状況

`project_assets.projects[].claude_md`と`global_assets.assets.claude_md`をもとに、ログに登場したプロジェクトの現在の`CLAUDE.md`と、Claude Codeのグローバル設定に現在存在する`CLAUDE.md`を分析する。

必ず以下を明示する。

```text
この分析は現在のファイル状態に基づくものであり、過去セッション時点で同じ内容が存在していたことは保証しません。
```

プロジェクト固有の`CLAUDE.md`とグローバル`CLAUDE.md`は、役割を分けて評価する。

見る観点:

- グローバルに置くべき横断的な運用ルールがあるか
- プロジェクト固有に置くべき技術スタック、検証コマンド、制約があるか
- 同じ指示がグローバルとプロジェクトで重複していないか
- 両者が矛盾していないか

### 9. 現在のSKILLS整備状況

`project_assets.projects[].skills`と`global_assets.assets.skills`をもとに、ログに登場したプロジェクトの現在のSkill整備状況と、Claude CodeのグローバルSkill整備状況を分析する。

見る観点:

- Skillの有無
- Skill名とdescriptionの明確さ
- 使用条件の明確さ
- ワークフローの具体性
- 似た責務のSkillが重複していないか
- グローバルSkillとして置くべき汎用Skillと、プロジェクトSkillとして置くべき固有Skillが分かれているか

### 10. 次回から使うプロンプトテンプレート

分析結果をもとに、次回から使えるテンプレートを複数提示する。

最低限、以下を含める。

- デバッグ依頼テンプレート
- 実装依頼テンプレート
- レビュー依頼テンプレート
- 短文承認テンプレート

### 11. 次の1週間で試す運用改善

実行しやすい改善策を3から5個に絞って提示する。

各改善策は、具体的な行動として書く。

## トーン

- 率直に書く。
- ただし断定しすぎない。
- ユーザーの能力評価ではなく、Claude Codeとの協働改善として書く。
- 問題点よりも、次にどう依頼すればよいかを重視する。
